## ⏱️ 一、时间与日期

- **所有时间统一使用北京时间 (UTC+8)**，标注为"北京时间"
- 文档生成时间也标注北京时间

### ESPN API 时区（🚨 每次拉数据前必读）

> ESPN API 返回的 `date` 字段是 UTC 时间（如 `2026-06-24T19:00Z`），**不是北京时间**。
> `scoreboard?dates=20260624` 返回的是美国当地日期，与北京有8小时时差。
> **禁止用 `dates=` 参数按北京日期筛选。**

**每次拉取赛程/比分的标准代码：**

```python
import urllib.request, json
from datetime import datetime, timedelta

url = 'https://site.api.espn.com/apis/site/v2/sports/soccer/fifa.world/scoreboard'
req = urllib.request.Request(url, headers={'User-Agent': 'Mozilla/5.0'})
j = json.loads(urllib.request.urlopen(req, timeout=15).read())

for evt in j.get('events', []):
    utc_str = evt.get('date', '')           # "2026-06-24T19:00Z"
    utc_dt = datetime.fromisoformat(utc_str.replace('Z', '+00:00'))
    bjt_dt = utc_dt + timedelta(hours=8)    # 北京时间
    bjt_date = bjt_dt.strftime('%Y-%m-%d')  # 如 "2026-06-25"
    bjt_time = bjt_dt.strftime('%H:%M')     # 如 "03:00"
    # 按 bjt_date 分组，不是 ESPN 返回的日期
```

| 字段 | 含义 | 示例 |
|------|------|------|
| `evt['date']` | UTC 时间 | `2026-06-24T19:00Z` |
| 换算后 | 北京时间 | `2026-06-25 03:00` |

- `status.type.state`: `'pre'`=未开始, `'in'`=进行中, `'post'`=已结束

## 🔬 五、分析铁律

### 1. 亚非球队不低估
面对强队时，亚洲/非洲球队防守纪律往往更强。不做5项韧性评估 = 不合格。
→ 案例: 英格兰0-0加纳，加纳5/5韧性满分

### 2. 强队三类分法

> 三类分法的目的是预测强队面对大巴时的破局能力。

| 类型 | 定义 | 破大巴能力 | 预测倾向 |
|------|------|-----------|---------|
| **超级巨星型** | 有≥2名具备"单兵破局"能力的球员，且均有本赛季顶级联赛验证 | 强 — 个人能力可强行破局 | 可高看大胜 |
| **体系型** | 进攻依赖整体配合，核心球员一旦被锁即无第二方案 | 弱 — 大巴是最致命的克制 | 防小胜/防平局 |
| **低效热门** | 纸面强但近期状态差、更衣室乱、或历史数据不支持 | 取决于具体问题 | 防冷平 |

**判定标准（用数据，不凭感觉）:**

> 🚨 每次预测前必须查 Transfermarkt 身价数据。**禁止凭感觉判断"纸面强"**。

**球员分级（基于 Transfermarkt 市场价值）:**

| 级别 | 身价阈值 | 能力标签 | 示例 |
|:---:|------|------|------|
| **超巨** | ≥€80M | 单兵可改变比赛 | 姆巴佩(€200M)、亚马尔(€200M)、哈兰德(€200M)、维尼修斯(€180M)、萨拉赫(€55M—年龄折扣但仍是超巨) |
| **核心** | €40M–€80M | 好球员但非不可替代 | 巴尔韦德(€90M)、佩德里(€80M)、特罗萨德(€45M) |
| **轮换** | €15M–€40M | 体系依赖型 | — |
| **角色球员** | <€15M | 纯体系球员 | 亚洲/非洲/大洋洲球队主力 |

**强队三类分法（结合身价）:**

- 条件A: 阵容总身价 >€500M（或对手的5倍以上）
- 条件B: ≥2名超巨且在不同进攻方式（如边锋≠中场≠中锋）
- A+B = **超级巨星型** → 可高看大胜
- A+¬B = **体系型** → 防小胜/防平局（对方只需锁死1人）
- ¬A+¬B = **低效热门** → 防冷平
- 示例: 葡萄牙(C罗+B费+莱昂) = 超级巨星型 → 5-0。英格兰(凯恩1人+贝林厄姆不稳定) = 体系型 → 0-0。巴西(维尼修斯+库尼亚+内马尔) = 超级巨星型。
- 示例: 比利时(€530M, 超巨仅德布劳内€55M) = 体系型或低效热门

**🚨 阵容身价比规则（防止比利时5-1式错误）:**

| 身价比 | 含义 | 预测约束 |
|------|------|------|
| >20:1 | 极端差距 | 🚨 **强队方不做0-0预测**。弱队除非已在本届验证过大巴(如佛得角0-0西班牙) |
| 10:1 – 20:1 | 大差距 | 强队正常发挥即胜。不做"1-1平局"首选 |
| 3:1 – 10:1 | 明显差距 | 强队占优但大巴/定位球可翻 |
| <3:1 | 接近 | 任何结果都可能 |

> **案例(6/27)**: 比利时€530M vs 新西兰~€18M ≈ 29:1 → 赛前预测1-1 = **严重违反身价比规则**。实际5-1。
> **案例(6/27)**: 西班牙€1.22B vs 乌拉圭€359M ≈ 3.4:1 → 西班牙胜方向正确。
> **案例(6/16)**: 西班牙€1.22B vs 佛得角~€50M ≈ 24:1 → 0-0 = **例外**——但佛得角已在本届验证过0-0西班牙，属"已证明大巴"例外。

**不要用对阵大巴强队的表现外推弱队：** 强队对伊朗10人5-4-1 0-0 ≠ 强队对新西兰4-2-3-1(必须赢)也会0-0。弱队不摆大巴时，身价比差距会兑现。」

### 10. 天气因子必须写条件分支（🚨 7/6巴西vs挪威教训）

> 禁止将天气因子作为确定性因子提前一天锁死。

**案例(7/6)**: 预测巴西vs挪威前，基于天气预报"体感41°C"写入"高温→巴西★★"，并花了大量篇幅做科研论证(Kyklos 2024+EJSS 2025+Schwarz 2025)。**实际当天阴天**，体感远低于41°C——整个高温因子全部作废，科研论证白费。挪威防线在阴天条件下组织完好，巴西0运动战进球。最终挪威2-1爆冷。

**强制规则:**
- 天气因子必须写成 **「如果...则...」** 条件分支，不可写为确定性方向
- 示例: 「如高温兑现→巴西有利(防守松散化, 有科研支撑)；如阴天/凉爽→此因子作废」
- 预测前24小时以外看到的天气预报 = 方向性提示，不是确定性输入
- 赛前更新时: **天气确认为阴天/凉爽/非极端→原高温因子直接删除，不可降级保留**

### 11. 刚伤愈→首发必须标注犯规+再伤双风险（🚨 7/6墨西哥vs英格兰教训）

> 刚伤愈的球员在非自然位置首发时，铲球精度和身体控制显著下降。

**案例(7/6)**: 英格兰RB匡萨。原预测"脚踝伤→缺席"。赛前更新发现他训练通过、图赫尔确认"fully available"，给了他"防守改善★"评级。实际: 54'高位铲球→VAR红牌。ESPN描述为"mistimed lunge"。**脚踝刚愈→铲球精度↓→红牌风险→10人应战**。这是比赛的#1转折点。

**强制规则:**
- 任何标注"刚伤愈→首发"的球员，必须在伤病表和分析中**同时标注**:
  - ① **[犯规风险↑]** — 伤后身体控制下降→铲球/对抗犯规概率↑
  - ② **[再伤风险↑]** — 同一部位再次受伤的概率高于正常
- 刚伤愈+非自然位置(如CB打RB)= **双重风险**，必须红色标注
- 不可仅写"防守改善XX"而忽略风险端

### 12. "N场零封"需要对手攻击力加权（🚨 7/6墨西哥vs英格兰教训）

> "N场零封"的含金量完全取决于对手攻击力。不可把对弱队的零封外推到对超巨型强队。

**案例(7/6)**: 墨西哥4场零封(南非/韩国/捷克/厄瓜多尔)。原预测给予"墨西哥★★★"。实际: 英格兰**仅用36分钟**就终结了360+分钟的不失球纪录。贝林厄姆98秒双响。这4场零封的对手无一是顶级攻击力。

**强制规则:**
- 写"N场零封"因子前，必须列出被零封的对手名单及其攻击力评级
- 若N场均面对非超巨型对手 → 因子星级**自动降一级**
- 格式: 「墨西哥4场零封 → 对手: 南非(T4)/韩国(T3)/捷克(T3)/厄瓜多尔(T3) → 从未面对超巨型 → 降为★★」
- 不可仅写数字而忽略对手质量

### 13. "替补可用" ≠ 因子贡献 — 未出场=0（🚨 7/6巴西vs挪威教训）

> 标注"替补可用"的球员只有在实际出场时才贡献价值。

**案例(7/6)**: 巴西vs挪威。原预测"拉菲尼亚回归替补=巴西★★"。实际: **拉菲尼亚未出场**。

**强制规则:**
- 替补球员不可单独作为因子表的一行
- 如需提及 → 写入教练博弈的"替补改变比赛能力"栏，标注「若出场则...」
- 因子表只列确定性因素（首发球员、已确认的战术安排）

### 14. 高原因子降级: ★★★ → ★★（🚨 7/6墨西哥vs英格兰教训）

> 理论上的"60分钟后体能断崖"可被心理韧性+超级巨星能力抵消。

**案例(7/6)**: 墨西哥vs英格兰 海拔2240m。原预测高原因子"墨西哥★★★"。实际: 英格兰不仅上半场36-38分钟连进2球(体能最佳证明)，而且在10人应战(54'红牌后)守住27分钟+11分钟补时=38分钟少一人防守→**体能未出现断崖**。BBC: 贝林厄姆补时阶段仍有体能铲断。

**强制规则:**
- 高原因子从★★★降为★★ — 保留为墨西哥利好，但不再视为"决定性因子"
- 当强队有≥3名超巨(凯恩+贝林厄姆+萨卡+赖斯=4人)时，高原因子可进一步降为★
- 高原≠不可战胜。超巨星密度是最有效的抗高原武器

### 15. 淘汰赛1-0必须列为高概率独立比分选项（🚨 7/7葡萄牙vs西班牙教训）

### 17. 禁止将裁判异常比赛外推为"破大巴困难"通用证据（🚨 7/10法国vs摩洛哥预测分析）

### 18. "N场零封"必须逐场解剖对手攻击力（🚨 7/11西班牙vs比利时复盘）

**S4 详细说明 — 身价≠当前能力的三种失真:**

> Transfermarkt 身价反映转会市场价值（年龄+合同+潜力+商业价值），**不是球员当前场上能力的直接衡量**。我们的强队三类分法使用身价作为能力代理——这在大多数情况下成立，但在两种极端上完全失效。

| 失真类型 | 特征 | 身价反映什么 | 场上实际能力 | 案例 |
|------|------|------|------|------|
| **类型1: 年龄折扣型** | ≥33岁，曾是世界级 | 零或接近零（无人会付转会费） | **特定能力仍存**（头球、位置感、传球视野、门将扑救）但速度/带球推进/压迫能力已退化 | C罗(€3-5M: 头球+终结仍威胁, 但0过人0推进) / 德布劳内(€55M: 创造力仍世界级, 但34岁体力存疑) / 库尔图瓦(€25M: 扑救仍是世界级) |
| **类型2: 年轻潜力型** | ≤21岁，天赋极高但产出未兑现 | **未来潜力**（3年后可能成为世界最佳） | 当前产出≠身价。高身价≠本届表现好 | 亚马尔(€200M: 17次过人, 但1球0助肌腱伤后未达100%) / 库巴尔西(€80M: 19岁球商高, 但面对伪9经验不足) |
| **类型3: 双失真叠加** | 双方同时存在类型1和类型2 | 身价比数字严重失真 | 场上实际对位差距可能与身价差方向相反 | C罗(€3M) vs 库巴尔西(€80M)=身价27:1→但C罗禁区威胁>身价暗示 / CDK(€40M) vs 库巴尔西(€80M)=身价1:2→但CDK伪9威胁>身价比暗示 |

**S4 强制检查（评估零封对手攻击力时）：**

```
对每一个被零封对手的核心攻击手，回答三个问题:

[ ] 他的身价中，有多少是"未来潜力"？→ 他本届的实际产出是多少？(进球? 助攻? 过人?)
[ ] 用"当前场上能力"而非"身价"排序 → 他对本场防线的实际威胁级别是多少？
```

**♟ 与强队三类分法的关系：** 强队三类分法使用身价作为"超巨密度"代理——仍然是有效框架。但评估时必须同时考虑失真类型：

- 评估**被零封对手攻击力** → 穿透身价（类型1: C罗€3M≠他的禁区威胁为零）
- 评估**本场对手攻击力** → 穿透身价（类型2: 亚马尔€200M≠他本届能单兵破局）

- "零封纪录"是一个数字，不是防守质量。防守质量 = 被零封的对手攻击力总和。

#### 19a. "大巴"标签必须用数据验证（🚨 7/12阿根廷vs瑞士QF教训）

> 给任何球队贴"大巴"标签前，必须用过程数据（控球率/射门/传球模式）验证，不能用结果数据（零封比分）推断。

**案例(7/12)**: 瑞士vs阿根廷。我们给瑞士5/5大巴满分（低位防守/前30分钟均★★★★★），基于"连续2场零封(阿尔及利亚/哥伦比亚)"。实际瑞士53.9%控球率+9射门5射正——不仅不是大巴，而且比阿根廷更主动进攻。错误根因: 用"0-0比分"这个结果推断战术类型，而非用过程数据判断。

**"大巴"判定标准（三项满足≥2项才能贴标签）:**

| # | 问题 | 数据来源 | 大巴阈值 | 本场示例(瑞士) |
|:---:|------|------|:---:|------|
| 1 | 本届平均控球率是否<40%？ | ESPN API 统计（检查前3-5场） | <40% = 大巴信号 | **53.9%** → ❌ 不满足 |
| 2 | 本届场均射门是否<8次？ | ESPN API | <8次 = 大巴信号 | **9次** (仅QF) → ❌ 不满足 |
| 3 | 淘汰赛是否主动放弃控球？ | 最近1-2场boxscore | 控球<35% + 射门<对手1/3 = 大巴信号 | QF vs阿根廷控球53.9% → ❌ 不满足 |

> **三项无一满足 → 不可贴"大巴"标签。** 瑞士0/3 = 不是大巴球队。

**强制规则:**
- 每次给对手贴"大巴""低位防守""深度回撤"等标签前 → 必须跑上述三项检查
- 三项中≥2项满足 → 可贴大巴标签，但须注明数据来源
- 三项中<2项满足 → 禁止贴大巴标签。改称"防守组织好"或"淘汰赛保守"
- **"零封"="大巴"是逻辑跳跃** — 零封是结果（可能来自控球优势/门将神扑/对手射门差），大巴是战术选择（主动放弃控球+深度回撤+限制射门数）

**半决赛适用性声明:** 四强球队（法国/西班牙/英格兰/阿根廷）均为T0-T1顶级攻击力球队，本届控球率均>45%，场均射门均>12次。不存在大巴球队。半决赛预测中不使用"大巴"框架分析。

### 3. 禁止绝对化
禁用"必胜""稳赢""一定打穿""毫无疑问""不可能输"
→ 替代: "更倾向于""大概率""首选方向"
每个结论附: 主要不确定性 + 潜在冷门路径

### 4. 换人必须双面评估
边翼卫/后卫换下时，同时评估两问:
- ① 进攻贡献(传中/定位球/反击)是否消失？
- ② 替补者的防守是否不如被换下者？
→ 案例: 6/24马苏阿库被换下，分析只说①(定位球移除利好哥伦比亚)，漏了②(左路防守减弱→穆尼奥斯76'右路破门)。且分析写利好，预测却写0-0 = 自相矛盾。

### 5. 比赛独立分析
禁止用一场比赛的结果推导同天另一场。不存在"X为全天定调"。
→ 案例: 6/24英格兰0-0加纳 ≠ 哥伦比亚也会0-0

### 6. 教训必须数据先行
先跑数据验证假设，再写教训。不要先写教训再找数据。
无可靠数据支撑的"教训" → 写成"待验证假设"。

### 7. 复盘与预测必须分开输出

> 复盘和预测是两个完全不同类型的任务，合在一起时每个都做不够好。

| 规则 | 说明 |
|------|------|
| **禁止合写** | 一个请求里同时要复盘+预测时，分两次输出。先写复盘文件，再写预测文件。 |
| **禁止缩水** | 如果任务量太大写不完完整框架 → 直接说"这个太长，我先写A，B放在下一轮"，不要偷偷减量。 |
| **预测必须10项** | 每场比赛必须完整覆盖10项必填清单，不得因场次多而跳过任何一项。 |
| **复盘必须逐场** | 每场比赛必须逐场对比原始预测 vs 实际结果，附教训。不得合并成一段话。 |
| **先声明再分步** | 6场以上任务 → 先告诉用户"我会分X轮完成"，然后每轮完成后再开始下一轮。 |

→ 案例: 6/26复盘+6/27预测合在一份文件→预测部分框架缩水(缺冷门路径/韧性评估/定位球)→被用户指出。
→ 正确做法: 先输出"6月26日复盘.md"(逐场对比+教训)，再单独输出"6月27日预测.md"(完整10项×6场)。

### 🚨 FIFA 官方 API（实时比赛事件 — 首选数据源）

> **免费、无需 API Key、公开访问。** 一次调用返回全部事件（进球/换人/黄红牌/阵容/控球率/比赛时间）。

```python
import urllib.request, json

# 标准调用
url = 'https://api.fifa.com/api/v3/live/football/{match_id}?language=en'
req = urllib.request.Request(url, headers={'User-Agent': 'Mozilla/5.0'})
data = json.loads(urllib.request.urlopen(req, timeout=15).read())

# 提取换人/进球/黄红牌
for side, team in [('HOME', data['HomeTeam']), ('AWAY', data['AwayTeam'])]:
    for g in team.get('Goals', []):
        print(f"⚽ {g['Minute']} {g['PlayerName'][0]['Description']}")
    for s in team.get('Substitutions', []):
        print(f"🔄 {s['Minute']} {s['PlayerOnName'][0]['Description']} IN / {s['PlayerOffName'][0]['Description']} OUT")
    for b in team.get('Bookings', []):
        card = '🟥' if b.get('CardType') == 2 else '🟨'
        print(f"{card} {b['Minute']} {b['PlayerName'][0]['Description']}")
    print(f"⚽ Goals: {len(team.get('Goals',[]))} | 🔄 Subs: {len(team.get('Substitutions',[]))} | 🟨 Bookings: {len(team.get('Bookings',[]))}")
```

| 字段 | 含义 | 示例 |
|------|------|------|
| `MatchTime` | 当前比赛分钟 | `"78'"` |
| `MatchStatus` | 0=未开始, 1=上, 3=中场, 5=下, 10=结束 | `5` |
| `HomeTeam.Score` / `AwayTeam.Score` | 当前比分 | `0` / `1` |
| `Goals[].Minute` | 进球时间 | `"42'"` |
| `Substitutions[].Minute` | 换人时间 | `"60'"` |
| `Substitutions[].PlayerOffName[0].Description` | 被换下球员 | `"PEDRI"` |
| `Substitutions[].PlayerOnName[0].Description` | 换上球员 | `"Fabian RUIZ"` |
| `Bookings[].CardType` | 1=黄牌, 2=红牌 | `1` |
| `Substitutions[].Reason` | 4=受伤, 0=战术 | `4` |
| `BallPossession` | 控球率对象 | — |

**match_id 获取：**
- 从 FIFA 赛程日历: `https://api.fifa.com/api/v3/calendar/matches?date=2026-06-27` → 返回中 `IdMatch` 字段
- 已知ID: `400021484`=乌拉圭vs西班牙, `400021485`=佛得角vs沙特阿拉伯
- 从 FIFA 赛事中心 URL 提取: `fifa.com/en/match-centre/match/17/285023/289273/{match_id}`

**🚨 注意:**
- 球员名为英文全大写（如 `PEDRI`），需对照 Players 数组中的 `ShirtNumber` 确认译名
- ESPN API 的 details 不返回换人 → FIFA API 是唯一直接返回换人的接口
- 比 ESPN API 快 ~30 秒（FIFA 官方数据更近源头）

### 10.1 anysearch — 搜索规范

```bash
# 通用搜索
Skill("anysearch", "search \"query\" --max_results 5")

# 垂直搜索（金融/体育等→先 get_sub_domains）
Skill("anysearch", "get_sub_domains --domain sports")

# 批量并行搜索
Skill("anysearch", "batch_search --query \"A\" --query \"B\" --max_results 5")

# 提取全文
Skill("anysearch", "extract \"https://...\"")
```

**何时用 anysearch 而非 WebSearch/WebFetch:**
- anysearch 是项目指定搜索工具，结果更结构化
- WebSearch/WebFetch 仅作为 anysearch 不可用时的**最后备选**
- 每次搜索前先用 anysearch，不要跳过去用 WebSearch

### 10.2 docx — 文档生成规范

**🚨 每次写完预测 .md 后，必须立即用 docx skill 生成对应的 .docx 文件。**

调用格式：
```
Skill("docx", "根据 E:\ai\世界杯\2026年X月X日_两场预测.md 的内容生成正式预测报告 docx。要求: 横向A4、微软雅黑9-10pt、深蓝表头白字、隔行灰底、首选行浅绿高亮、三位国家缩写替代国旗、[对][错][待]替代emoji。输出路径: E:\ai\世界杯\2026年X月X日_两场预测.docx")
```

**docx skill 内部使用 `docx-js` (JavaScript) 生成，不是 python-docx。** 生成后自动运行 `validate.py` 验证。

**禁止的做法:**
- ❌ 手写 `gen_docx.py` 用 python-docx 生成 → 这是绕开 skill
- ❌ 生成完 .md 就结束 → 必须接着生成 .docx
- ❌ 用 emoji 国旗和符号 → docx 中必须用三位缩写和 [对][错][待]

### 10.3 复盘 .docx 也要生成

复盘文件写完后，同样用 `docx` skill 生成对应的 .docx 复盘报告。
